## 1. Overview

Phase 2 introduces SQLAlchemy 2.0 as the ORM layer and Alembic for schema migrations. It does **not** replace the existing raw-SQL `db.py` methods — that's Phase 3. This phase defines ORM models against the Phase 1 DDL, autogenerates the initial Alembic migration, sets up session management, and migrates `init_db()` to use `Base.metadata.create_all()`. The raw SQL CRUD in `db.py` currently targets the **pre-Phase 1 schema** and will fail against the new database (columns and tables were removed in Phase 1). This is expected — Phase 3 will rewrite them. Phase 2 only changes `init_db()` to use ORM; the rest of `db.py` is left untouched. SQLAlchemy and raw `sqlite3` can coexist structurally against the same database.

## 2. Design Decisions (Restated from Roadmap)

### 2.1 Separate ORM models from Pydantic API models

`quarry/store/models.py` holds SQLAlchemy `Mapped[]` classes (database schema).  
`quarry/models.py` holds Pydantic `BaseModel` subclasses (API/validation boundaries).

Conversion: `PydanticModel.model_validate(orm_obj, from_attributes=True)`.

**Why not SQLModel?** SQLModel combines ORM + Pydantic into a single class, forcing 1:1 DB-to-API mapping. Phase 1 already broke that mapping — fields moved from `Company` to `UserWatchlistItem`, from `JobPosting` to `UserSimilarityScore`. Separate models preserve flexibility.

### 2.2 Session management

A single `Engine` + `sessionmaker` pattern with `PRAGMA foreign_keys = ON` enforced on every connection via SQLAlchemy event listener.

```python
engine = create_engine("sqlite:///quarry.db")
SessionLocal = sessionmaker(bind=engine)

@event.listens_for(engine, "connect")
def _set_pragma(dbapi_connection, connection_record):
    cursor = dbapi_connection.cursor()
    cursor.execute("PRAGMA foreign_keys = ON")
    cursor.close()
```

This is critical: SQLite starts with foreign keys off by default, and SQLAlchemy does not auto-enable them.

### 2.3 Alembic for migrations

The initial migration is generated via `alembic revision --autogenerate -m "initial multi-user schema"` and then **diffed against the Phase 1 raw DDL** to verify no loss. All future schema changes go through `alembic revision --autogenerate`.

### 4.2 `quarry/store/models.py` — All ORM Models

This is the largest single file in Phase 2. Must define exactly one class per table, with:

- `Mapped[]` type annotations on every column
- `relationship()` with `back_populates` for shared↔per-user links
- `__table_args__` for `UniqueConstraint`, `Index`, `CheckConstraint`
- `ForeignKey(ondelete="CASCADE")` matching the raw DDL

**Important:** `model_version_id` in `UserClassifierScore` uses `ForeignKey(ondelete="SET NULL")` — the only non-CASCADE FK in the schema.

Full file listing follows. Column definitions must exactly match the Phase 1 DDL.


### 4.3 `quarry/store/session.py` — Engine & Session Factory

```python
# quarry/store/session.py
"""SQLAlchemy engine and session management.

Provides:
- get_engine(): create or return the singleton SQLite engine
- get_session(): return a new Session bound to the engine
- session_scope(): context manager for transactional sessions
- PRAGMA foreign_keys = ON enforced on every connection
"""
from contextlib import contextmanager
from pathlib import Path
from typing import Generator

from sqlalchemy import Engine, create_engine, event
from sqlalchemy.orm import Session, sessionmaker

from quarry.store.models import _pragma_foreign_keys_on


_engines: dict[str, Engine] = {}
_SessionLocal: sessionmaker | None = None


def get_engine(db_path: str | Path | None = None) -> Engine:
    """Return (or create) a SQLAlchemy Engine for the given database path.

    Engines are cached per database URL so tests using tmp_path
    get isolated engines per test database. SQLite requires a single
    writer per database, so a singleton per URL is appropriate.

    Args:
        db_path: Path to SQLite database. Uses config if not provided.

    Returns:
        SQLAlchemy Engine configured for the project database.
    """
    global _SessionLocal

    if db_path is None:
        from quarry.config import settings

        db_path = settings.db_path

    db_url = f"sqlite:///{db_path}"

    if db_url not in _engines:
        engine = create_engine(
            db_url,
            echo=False,
        )
        event.listen(engine, "connect", _pragma_foreign_keys_on)
        _engines[db_url] = engine

    return _engines[db_url]


def get_session() -> Session:
    """Return a new SQLAlchemy Session bound to the current engine.

    The caller is responsible for closing the session.
    Prefer session_scope() for automatic cleanup.
    """
    global _SessionLocal
    engine = get_engine()
    # Recreate sessionmaker if engine changed (e.g., tests with different DBs)
    if _SessionLocal is None or _SessionLocal.kw["bind"] is not engine:
        _SessionLocal = sessionmaker(bind=engine)
    return _SessionLocal()


@contextmanager
def session_scope(engine: Engine | None = None) -> Generator[Session, None, None]:
    """Context manager for transactional session scope.

    Commits on success, rolls back on exception.

    Usage:
        with session_scope() as session:
            session.add(Company(name="Acme"))

    Args:
        engine: If provided, uses this engine directly (e.g., for init_db
                against a custom path). If None, uses get_session() which
                binds to the default engine from config.
    """
    if engine is not None:
        session = Session(bind=engine)
    else:
        session = get_session()
    try:
        yield session
        session.commit()
    except Exception:
        session.rollback()
        raise
    finally:
        session.close()
```

**Design notes:**

- The engine is a module-level singleton. Multiple engines against the same SQLite file can cause "database is locked" errors. Since the existing `Database` class opens/closes connections per operation, the ORM's connection pooling (enabled by default) coexists fine.
- We do NOT use `check_same_thread=False` in the engine connect args. With a single writer, the default is correct. If we later add multi-threaded access (e.g., APScheduler + Flask), SQLite WAL mode should be enabled instead.
- `session_scope()` is provided for Phase 3, but Phase 2 does not rewrite CRUD methods. It's included here so the `init_db()` migration can use it.

```python
def init_db(db_path: str | Path) -> Database:
    """Initialize database with schema using SQLAlchemy create_all.

    Uses checkfirst=True so it's idempotent — tables are only created
    if they don't exist. The raw SQLite3 Database object is still returned
    for Phase 2 backward compatibility (Phase 3 will change this).
    """
    from quarry.store.models import Base
    from quarry.store.session import get_engine, session_scope

    db_path = Path(db_path)

    # Create tables via SQLAlchemy ORM metadata
    engine = get_engine(db_path)
    Base.metadata.create_all(engine, checkfirst=True)

    # Seed default user (id=1) if not present
    with session_scope(engine=engine) as session:
        from quarry.store.models import User
        from sqlalchemy import select

        existing = session.execute(
            select(User).where(User.id == 1)
        ).scalar_one_or_none()
        if existing is None:
            session.add(User(
                id=1, email="default@local", name="Default User"
            ))

    # Return the existing raw-sqlite3 Database for backward compat
    db = Database(db_path)
    return db
```

**Key changes:**

1. Import `Base` from ORM models (not `SCHEMA_SQL` from schema.py)
2. `create_all(checkfirst=True)` instead of `executescript(SCHEMA_SQL)` — idempotent
3. Default user seed is now handled separately (ORM's `create_all` creates tables only, no data)
4. Still returns a raw `Database` object — callers are unchanged

**Edge case: two init paths producing identical databases.** The raw DDL and `create_all` must produce exactly the same schema. The diff verification (Section 6.3) confirms this. If autogenerate finds discrepancies, fix the ORM model definitions, not the migration.

```python
# alembic/env.py
import os
import sys
from logging.config import fileConfig

# Ensure repo root is in sys.path so quarry imports resolve
sys.path.insert(0, os.path.dirname(os.path.dirname(os.path.abspath(__file__))))

from alembic import context
from sqlalchemy import create_engine, pool, text

# Import our ORM models so Base.metadata is populated
from quarry.store.models import Base

# this is the Alembic Config object
config = context.config

# Interpret the config file for Python logging
if config.config_file_name is not None:
    fileConfig(config.config_file_name)

# Set target_metadata to our ORM Base
target_metadata = Base.metadata

# Dynamically set sqlalchemy.url from project config
def get_url() -> str:
    from quarry.config import settings
    return f"sqlite:///{settings.db_path}"

def run_migrations_offline() -> None:
    """Run migrations in 'offline' mode."""
    url = get_url()
    context.configure(
        url=url,
        target_metadata=target_metadata,
        literal_binds=True,
        dialect_opts={"paramstyle": "named"},
    )
    with context.begin_transaction():
        context.run_migrations()

def run_migrations_online() -> None:
    """Run migrations in 'online' mode."""

    # Build engine directly with the configured URL (do not use
    # engine_from_config + URL reassignment — Engine.url is immutable)
    connectable = create_engine(get_url(), poolclass=pool.NullPool)

    with connectable.connect() as connection:
        # Enforce foreign keys during migration operations
        connection.execute(text("PRAGMA foreign_keys = ON"))

        context.configure(
            connection=connection,
            target_metadata=target_metadata,
        )
        with context.begin_transaction():
            context.run_migrations()

if context.is_offline_mode():
    run_migrations_offline()
else:
    run_migrations_online()
```

**Critical:** The `_pragma_foreign_keys_on` hook must be called during migration too. Alembic's connection bypasses the SQLAlchemy engine event system, so the pragma must be applied manually in `run_migrations_online`.

## 5. Edge Cases & Troubleshooting

### 5.1 `create_all` vs raw DDL discrepancies

If Phase 1 schema tests fail after switching `init_db()` to `create_all`, the ORM model definitions differ from the raw DDL. Common causes:

| Symptom                                       | Likely Cause                                      | Fix                                                |
| --------------------------------------------- | ------------------------------------------------- | -------------------------------------------------- |
| Missing CHECK constraint                      | `CheckConstraint` not in `__table_args__`         | Add to model class                                 |
| Wrong FK ondelete                             | `ForeignKey(ondelete=...)` missing or wrong value | Match raw DDL: CASCADE or SET NULL                 |
| Column nullable mismatch                      | `Optional[]` wrapper missing/extra                | Match `DEFAULT` vs `NOT NULL` from raw DDL         |
| Missing index                                 | `Index()` not in `__table_args__`                 | Add all indexes from raw DDL                       |
| `datetime.now()` vs `datetime.utcnow()`       | SQLAlchemy uses `datetime.now` by default         | Explicitly use `datetime.utcnow` in all defaults   |
| `create_all` creates tables, but no seed data | `create_all` only does DDL, not DML               | Seed default user separately (done in `init_db()`) |

### 5.2 Foreign keys not enforced during raw SQL operations

The `event.listen` hook in `session.py` enables foreign keys on every SQLAlchemy connection. However, **the raw `Database.get_connection()` in `db.py` has its own** `PRAGMA foreign_keys = ON` — it uses `sqlite3.connect()` directly and does NOT go through the SQLAlchemy engine. This means:

- ORM operations through `session_scope()`: foreign keys enforced ✅
- Raw `Database.execute()` calls: foreign keys enforced ✅ (existing code already sets pragma)
- Alembic migrations: foreign keys enforced ✅ (explicit call in `run_migrations_online`)

### 5.3 `engine` singleton and test isolation

Engines are cached per database URL (via `_engines: dict[str, Engine]`). Each test using `tmp_path` gets its own engine pointing at its own SQLite file, because each `tmp_path` produces a different URL. After use, `engine.dispose()` releases the connection pool.

`get_engine()` without a path argument uses `quarry.config.settings.db_path` (production). Calling it in tests without a path would share the production engine — always pass `tmp_path`-scoped paths in test fixtures.

### 5.4 Alembic `--autogenerate` with CHECK constraints

SQLite CHECK constraints defined in `__table_args__` may not be detected by Alembic's autogenerate. This is a known SQLAlchemy dialect limitation:

- **Inline column CHECK** (e.g., `mapped_column(..., CheckConstraint(...))`): **not supported** in SQLAlchemy 2.0 for SQLite
- **Table-level CHECK** in `__table_args__`: **detected by autogenerate** ✅

All our CHECK constraints are table-level (in `__table_args__`), so this should work. If they're missing from the autogenerated migration, add them manually as described in Section 4.6.5.

### 5.5 `sqlalchemy.url` in `alembic.ini` vs env.py

Two approaches:

1. **Hardcoded in `alembic.ini`:** `sqlalchemy.url = sqlite:///quarry.db` — simple, but breaks if the path changes
2. **Dynamic in `env.py`:** Call `get_url()` to read from `quarry.config.settings.db_path` — more robust

**Recommendation:** Use approach (2) — dynamic from config. The `env.py` template in Section 4.6.3 implements this.

### 5.6 `Optional[]` type hints for nullable columns

SQLAlchemy 2.0 uses Python type hints to infer nullability:

- `Mapped[str]` → NOT NULL
- `Mapped[Optional[str]]` → NULL allowed
- `Mapped[str | None]` → NULL allowed (Python 3.10+)

The Phase 1 DDL has explicit `NOT NULL` (implicit for `PRIMARY KEY`, explicit on some TEXT columns). Match these exactly. If the hint says `Mapped[str]` but the raw DDL allows NULL, `create_all` will generate a NOT NULL column that breaks Phase 1 tests.

**Double-check from raw DDL:**

```sql
-- NOT NULL columns (Mapped[str], not Optional):
companies.name            TEXT NOT NULL
companies.ats_type        TEXT NOT NULL DEFAULT 'unknown'
companies.resolve_status  TEXT NOT NULL DEFAULT 'unresolved'
job_postings.company_id   INTEGER NOT NULL
job_postings.title        TEXT NOT NULL
job_postings.title_hash   TEXT NOT NULL
job_postings.url          TEXT NOT NULL
user_watchlist.user_id    INTEGER NOT NULL
user_watchlist.company_id INTEGER NOT NULL
user_posting_status.user_id  INTEGER NOT NULL
user_posting_status.posting_id INTEGER NOT NULL
user_labels.user_id       INTEGER NOT NULL
user_labels.posting_id    INTEGER NOT NULL
user_labels.signal        TEXT NOT NULL
user_search_queries.user_id  INTEGER NOT NULL
user_search_queries.query_text TEXT NOT NULL
user_similarity_scores.user_id INTEGER NOT NULL
user_similarity_scores.posting_id INTEGER NOT NULL
user_similarity_scores.similarity_score REAL NOT NULL
user_classifier_scores.user_id INTEGER NOT NULL
user_classifier_scores.posting_id INTEGER NOT NULL
user_classifier_scores.classifier_score REAL NOT NULL
```

All other columns are nullable (Optional).

**Boolean columns with defaults:** `active` (in `ClassifierVersion`, `UserWatchlistItem`, `UserSearchQuery`) and `is_active` (in `User`) use `Mapped[bool]` with Python-side `default=`. The raw DDL has `BOOLEAN DEFAULT TRUE/FALSE` with no explicit `NOT NULL`. `Mapped[bool]` infers `NOT NULL`, which is stricter than the DDL. This is acceptable — the DEFAULT makes NULL semantically unreachable — but will show as a discrepancy during diff verification (Section 4.6.5). To match exactly, use `Mapped[Optional[bool]]`.

### 5.7 Timestamp defaults

The raw DDL uses `DEFAULT CURRENT_TIMESTAMP` on timestamp columns. The ORM models use `server_default=func.now()` which generates the identical SQL. No Python-side `datetime` callable is needed — SQLite handles timestamps on INSERT. This also avoids the `datetime.utcnow()` deprecation in Python 3.12+.
