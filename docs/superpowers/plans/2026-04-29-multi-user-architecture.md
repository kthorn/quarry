# Multi-User Architecture — Phased Roadmap

**Status:** Draft

**Goal:** Rebuild the database and data access layer to cleanly separate shared catalog data from per-user preferences, then adopt SQLAlchemy 2.0 + Alembic as the ORM/migration layer before rewriting callers. The result: multi-user capable from day one, with a maintainable data layer and proper schema migrations.

**Related Documents:**

- `2026-04-29-multi-user-schema.md` — Phase 1 detailed schema DDL spec (this doc defers all DDL details there)
- `2026-04-28-schema-rebuild-multiuser.md` — Original comprehensive plan (superseded by this phased approach; retained as reference for caller-update specifics in Phase 4)

---

## Architecture Overview

```
┌─────────────────────────────────────────────────────┐
│                  Shared Catalog                     │
│  companies  job_postings  locations  crawl_runs     │
│  classifier_versions  agent_actions  system_settings│
│  (user-independent canonical data)                  │
└──────────────┬──────────────────────────────────────┘
               │ FK CASCADE
┌──────────────▼──────────────────────────────────────┐
│                 Per-User Data                       │
│  users  user_watchlist  user_posting_status         │
│  user_labels  user_search_queries                   │
│  user_similarity_scores  user_classifier_scores     │
│  user_enriched_postings  user_settings              │
│  (preferences, ratings, scores, config)             │
└─────────────────────────────────────────────────────┘
```

**Key insight:** Everything that depends on "who is asking" lives in per-user tables. Everything that's the same regardless of who asks lives in shared tables. Defaults to `user_id=1` until auth is added.

## Phase 1: Ship Raw DDL Schema

**Scope:** `quarry/store/schema.py` only  
**Spec:** `2026-04-29-multi-user-schema.md` (full DDL, column migration table, test plan)  
**Entry criterion:** Current DB backed up  
**Exit criterion:** New `quarry.db` committed with all new tables, schema tests pass

This phase delivers the final schema design as raw SQL. No code changes to `db.py`, `models.py`, or any caller — just the DDL and tests that verify it. **Existing CRUD tests will fail** (the old `Company(active=True)` and `JobPosting(status='new')` don't map to the new schema); this is expected and resolved in Phase 3.

**Why raw SQL first:** The schema is complex enough to review on its own. Shipping the DDL as SQL means we can verify CHECK constraints, FK cascades, and indexes without an ORM translation layer in the way. The DDL also becomes the target for Alembic's first autogenerated migration in Phase 2 — we diff to confirm nothing was lost.

### Tasks

1. Rewrite `SCHEMA_SQL` in `quarry/store/schema.py` (full DDL from schema spec)
2. Backup + delete old `quarry.db` from git
3. Run `python -m quarry.store init` to create new DB
4. Write schema verification tests (per-user isolation, FK cascades, CHECK constraints)
5. Remove `migrate_resolve_columns()` from `db.py` (dead code — new schema has those columns)
6. Commit

**See `2026-04-29-multi-user-schema.md` for the complete DDL, column migration table, and detailed task breakdown.**

## Phase 3: Rewrite CRUD Layer with ORM

**Scope:** `quarry/store/db.py` — replace raw SQL with SQLAlchemy queries  
**Prerequisite:** Phase 2 complete (ORM models and session management exist)

This is where the raw-to-ORM rewrite happens. The key principle: **method signatures stay the same (or add `user_id=1`), only the implementation changes.** Callers in Phase 4 should see minimal API surface changes.

### Pattern: Raw SQL → ORM

| Old (raw SQL)                                          | New (ORM)                                                     |
| ------------------------------------------------------ | ------------------------------------------------------------- |
| `self.execute("SELECT ... WHERE id = ?", (id,))`       | `session.get(Company, id)`                                    |
| `self.execute("INSERT INTO ... VALUES (?, ...)", ...)` | `session.add(Company(...)); session.flush()`                  |
| `self.execute("SELECT ... JOIN ... WHERE ...", ...)`   | `session.execute(select(Posting).join(Watchlist).where(...))` |
| Manual `cursor.lastrowid`                              | `session.flush(); obj.id`                                     |
| `self.executemany(sql, params)`                        | `session.execute(insert(Table), [...])`                       |
| `conn.executescript(SCHEMA_SQL)`                       | `Base.metadata.create_all(engine, checkfirst=True)`           |

### Key Method Rewrites

**`get_postings_with_scores`** — The most complex query. Replaces a 5-table hand-rolled JOIN with declarative relationship loading:

```python
def get_postings_with_scores(self, user_id: int = 1, status: str = "new",
                              limit: int = 20, offset: int = 0) -> list[dict]:
    from sqlalchemy import select, func

    stmt = (
        select(
            JobPosting,
            Company.name.label("company_name"),
            func.coalesce(UserPostingStatus.status, "new").label("status"),
            func.coalesce(UserSimilarityScore.similarity_score, 0.0).label("similarity_score"),
            func.coalesce(UserClassifierScore.classifier_score, 0.0).label("classifier_score"),
            func.coalesce(UserEnrichedPosting.fit_score, 0).label("fit_score"),
            UserEnrichedPosting.role_tier,
            UserEnrichedPosting.fit_reason,
        )
        .join(Company)
        .join(UserWatchlistItem, and_(
            JobPosting.company_id == UserWatchlistItem.company_id,
            UserWatchlistItem.user_id == user_id,
        ))
        .outerjoin(UserPostingStatus, and_(
            JobPosting.id == UserPostingStatus.posting_id,
            UserPostingStatus.user_id == user_id,
        ))
        .outerjoin(UserSimilarityScore, and_(
            JobPosting.id == UserSimilarityScore.posting_id,
            UserSimilarityScore.user_id == user_id,
        ))
        # ... remaining outerjoins for classifier_score, enriched
        .where(or_(
            UserPostingStatus.status == status,
            and_(UserPostingStatus.status == None, status == "new"),
        ))
        .order_by(func.coalesce(UserSimilarityScore.similarity_score, 0.0).desc())
        .limit(limit).offset(offset)
    )
    # Return list[dict] for template compatibility (Phase 4 may change this)
```

**`upsert_label`** — Replaces raw INSERT ON CONFLICT with session merge:

```python
def upsert_label(self, label: models.UserLabel) -> int:
    with self.session() as session:
        existing = session.execute(
            select(UserLabel).where(
                UserLabel.user_id == label.user_id,
                UserLabel.posting_id == label.posting_id,
                UserLabel.signal == label.signal,
            )
        ).scalar_one_or_none()
        if existing:
            existing.notes = label.notes
            existing.labeled_at = label.labeled_at or datetime.utcnow()
            existing.label_source = label.label_source
        else:
            orm_label = UserLabel(
                user_id=label.user_id,
                posting_id=label.posting_id,
                signal=label.signal,
                notes=label.notes,
                labeled_at=label.labeled_at or datetime.utcnow(),
                label_source=label.label_source,
            )
            session.add(orm_label)
            session.flush()
        session.commit()
```

**`mark_postings_seen`** — Bulk upsert via ORM:

```python
def mark_postings_seen(self, posting_ids: list[int], user_id: int = 1) -> None:
    if not posting_ids:
        return
    with self.session() as session:
        now = datetime.utcnow()
        for pid in posting_ids:
            session.execute(
                insert(UserPostingStatus).values(
                    user_id=user_id, posting_id=pid, status="seen",
                    first_seen_at=now, last_seen_at=now,
                ).on_conflict_do_update(
                    index_elements=["user_id", "posting_id"],
                    set_=dict(status="seen", last_seen_at=now),
                )
            )
        session.commit()
```

## Decisions & Tradeoffs

### Why sequential phases instead of one big PR?

**Debugging surface area.** Phase 1 isolates DDL correctness. Phase 2 isolates ORM configuration. Phase 3 isolates query translation. Phase 4 isolates caller plumbing. If something breaks, you know which phase introduced it. A single monolithic PR mixes all four concerns — "is this a schema bug, an ORM bug, a query bug, or a caller bug?" becomes impossible to answer quickly.

**Reversibility.** Each phase can be reverted independently. If Phase 2 (ORM) causes unexpected SQLite type quirks or session management pain, you can revert to Phase 1 and keep the raw SQL CRUD approach without losing the schema work. Similarly, if Phase 3's ORM queries are too slow, you can fall back to raw SQL for hot paths while keeping ORM for simple CRUD.

**Alembic baseline.** Running Phase 1's raw DDL first, then capturing it with `alembic revision --autogenerate`, is the standard workflow for existing projects adopting Alembic. The raw DDL serves as the reviewable spec; Alembic captures it as the authoritative migration.

### Why two model layers (ORM + Pydantic)?

The current codebase already uses Pydantic models that map 1:1 to DB columns — and the schema change _broke_ that mapping (fields moved from `Company` to `UserWatchlistItem`, from `JobPosting` to `UserSimilarityScore`, etc.). If the DB model and API model were already separate, only the DB model would need updating; the API model could maintain backward-compatible computed properties. This pattern also enables:

- Hiding internal columns from API responses (e.g., `embedding` BLOB)
- Computed fields that don't exist in the DB (e.g., `company_name` joined in)
- Different serialization shapes for different consumers (UI template vs CLI table vs API response)

The cost is ~18 model classes instead of ~9. Given that Pydantic models are declarative (zero logic), this isn't a real maintenance burden.

### Why not SQLModel?

SQLModel combines SQLAlchemy ORM and Pydantic into a single class. This looks appealing (one class instead of two) but forces the 1:1 DB-to-API mapping that we're deliberately breaking away from. The project already demonstrated that this mapping doesn't hold — fields moved between tables during this schema change. Separate models give us flexibility; SQLModel locks us into identity.
